    Block 1 — Overview

    What is JavaScript?

    HTML structures content. CSS styles it. JavaScript executes logic.

    Interpreted

    No compilation step — the engine reads and executes your code line by line.

    First-Class Functions

    Functions are values. You can store them in variables, pass them around, and return them.

    Single-Threaded

    One command at a time, one call stack. (Async patterns work around this, not against it.)

    Runs Everywhere

    Client-side (browser) and server-side (Node.js, Deno, Bun).

    One-liner: JavaScript is a versatile, cross-platform, object-oriented scripting language that powers both the web browser and the server.

    Block 1 — Types & Variables

    Data Types in JavaScript

    Primitive Types (immutable)

    let name    = "Alice";       // String
let age     = 25;            // Number
let active  = true;          // Boolean
let nothing = null;          // Null
let x;                        // Undefined
let id      = Symbol("id"); // Symbol — unique identifier, often used as object property keys
let big     = 9007199254740991n; // BigInt

    Non-Primitive Types (mutable, by reference)

    let arr  = [1, 2, 3];           // Array
let obj  = { name: "Bob" };     // Object
let fn   = function() {};       // Function
    Key difference: Primitives are stored by value. Non-primitives are stored by reference — two variables can point to the same object.
    Block 1 — Types & Variables

    Determining Data Types

    typeof — for primitives

    typeof "hello"    // "string"
typeof 42          // "number"
typeof true        // "boolean"
typeof undefined   // "undefined"
typeof null        // "object" ← famous JS bug!
typeof {}          // "object"
typeof []          // "object"
typeof function(){} // "function"

    instanceof — for non-primitives

    const arr = [1, 2, 3];
arr instanceof Array   // true
arr instanceof Object  // true

const obj = { a: 1 };
obj instanceof Object  // true
obj instanceof Array   // false
    Rule of thumb: Use typeof for primitives, instanceof for objects/arrays. And yes, typeof null === "object" is a 30-year-old bug that will never be fixed.
    Block 1 — Types & Variables

    Variables — var, let, const

    var x = 10;     // old way, function-scoped
let y = 20;     // modern, block-scoped, reassignable
const z = 30;   // modern, block-scoped, NOT reassignable

y = 25;          // OK
z = 35;          // TypeError: Assignment to constant
    Best practice: Default to const. Use let only when you need to reassign. Avoid var in new code.

    Naming Rules

    • Can contain letters, digits, _, $
    • Must start with a letter, _, or $
    • Case-sensitive: myVar ≠ myvar
    • Can’t use reserved keywords (let, const, function …)
    let userName  = "Alice";   // ✔ camelCase
let _private  = 42;        // ✔
let $element  = null;      // ✔
let 2fast     = "no";      // ✘ starts with digit
    Block 1 — Operators

    Operators

    Arithmetic & Assignment

    10 + 3    // 13
10 - 3    // 7
10 * 3    // 30
10 / 3    // 3.333...
10 % 3    // 1 (remainder)

let a = 5;
a += 3;   // a is now 8
a++;      // a is now 9

    Logical

    true && false   // false (AND)
true || false   // true  (OR)
!true            // false (NOT)

    Comparison — == vs ===

    5 ==  "5"    // true  (loose: coerces types)
5 === "5"    // false (strict: different types)
5 === 5      // true

0 ==  false  // true  (surprise!)
0 === false  // false (expected)

null ==  undefined // true
null === undefined // false
    Always use === (strict equality). The loose == operator coerces types in confusing ways and is a common source of bugs.

    Block 1 — Loops

    Loops

    for / while / do...while

    for (let i = 0; i < 3; i++) {
    console.log(i);  // 0, 1, 2
}

let n = 3;
while (n > 0) {
    console.log(n);  // 3, 2, 1
    n--;
}

do {
    console.log("runs at least once");
} while (false);

    for...of (arrays) vs for...in (objects)

    const colors = ["red", "green", "blue"];
for (const color of colors) {
    console.log(color);
}
// "red", "green", "blue" (values)

const person = { name: "Alice", age: 25 };
for (const key in person) {
    console.log(key, person[key]);
}
// "name" "Alice", "age" 25 (keys)
    Remember: for...of → values (arrays, strings). for...in → keys (objects).
    Block 2 — Functions

    Functions

    Function Declaration (Statement)

    function greet(name) {
    return "Hello, " + name;
}

console.log(greet("Alice"));
    Hello, Alice

    You can call a declared function even before it appears in the code. Expressions cannot do this.

    Function Expression

    // Named expression
const add = function addNumbers(a, b) {
    return a + b;
};

// Anonymous expression
const multiply = function(a, b) {
    return a * b;
};

console.log(add(3, 4));      // 7
console.log(multiply(3, 4)); // 12

    Expressions behave like regular variables — they must be defined before you call them.

    Block 2 — Functions

    Arrow Functions (ES6)

    A shorter syntax for writing functions. Introduced in ES6.

    Converting step by step

    // Regular function
const square = function(n) {
    return n * n;
};

// Arrow function
const square = (n) => {
    return n * n;
};

// Concise (single expression = implicit return)
const square = n => n * n;

    More examples

    // No parameters
const sayHi = () => console.log("Hi!");

// Multiple parameters
const add = (a, b) => a + b;

// Multi-line body (need braces + return)
const getUser = (name) => {
    const greeting = "Hello, " + name;
    return greeting;
};
    Arrow functions are concise and commonly used as callbacks, especially with array methods like map, filter, and reduce.
    Block 2 — Functions

    First-Class Functions

    In JavaScript, functions are first-class citizens — they can be used like any other value.

    Store in a variable

    const greet = (name) => `Hello, ${name}!`;
console.log(greet("Alice"));

    Pass as an argument

    function doTwice(fn) {
    fn();
    fn();
}
doTwice(() => console.log("Hi!"));
// "Hi!" "Hi!"

    Return from a function

    function multiplier(factor) {
    return (n) => n * factor;
}

const double = multiplier(2);
const triple = multiplier(3);

console.log(double(5));  // 10
console.log(triple(5));  // 15
    This is powerful! It means you can write functions that create other functions, and pass logic around like data.

    Arrays

    Creating & accessing

    Block 3 — Array Methods

    map, filter, reduce

    map — transform each item

    const nums = [1, 2, 3, 4];
const doubled = nums.map(n => n * 2);
// [2, 4, 6, 8]

    filter — keep items that match

    const evens = nums.filter(n => n % 2 === 0);
// [2, 4]

    forEach — just loop (no return)

    nums.forEach(n => console.log(n));

    reduce — accumulate into one value

    const sum = nums.reduce((acc, n) => acc + n, 0);
// 10
    How reduce works: 
    // Step by step:
// acc=0, n=1 → 0+1 = 1
// acc=1, n=2 → 1+2 = 3
// acc=3, n=3 → 3+3 = 6
// acc=6, n=4 → 6+4 = 10

    Chaining

    const result = nums
    .filter(n => n > 2)
    .map(n => n * 10);
// [30, 40]

    Block 3 — Scope

    Scopes — Where Do Variables Live?

    Variables are not visible everywhere. Their scope determines where they can be accessed.

    Global Scope

    Declared outside any function or block — accessible everywhere.

    const appName = "MyApp";
// accessible anywhere in the file

    Function Scope

    Declared inside a function — accessible only within that function.

    function greet() {
    var msg = "Hi";
    console.log(msg); // works
}
console.log(msg); // ReferenceError!

    Block Scope

    let and const are confined to the {} block. var leaks out.

    {
    let x = 10;
    const y = 20;
    var z = 30;
}
// x, y → ReferenceError
// z → 30 (var leaks!)
    Block 3 — Scope

    var vs let/const in Blocks & Shadowing

    Predict the output

    {
    var a = 10;
    let b = 20;
    const c = 30;

    console.log(a);  // ?
    console.log(b);  // ?
    console.log(c);  // ?
}

console.log(a);  // ?
console.log(b);  // ?
console.log(c);  // ?
    Inside block: 10, 20, 30
    Outside: a → 10 (var leaks out)
    b → ReferenceError (let is block-scoped)
    c → ReferenceError (const is block-scoped)

    Shadowing

    let x = "outer";

{
    let x = "inner";  // shadows the outer x
    console.log(x);   // "inner"
}

console.log(x);       // "outer" (unchanged)
    Shadowing means a variable with the same name in an inner scope "hides" the outer one. The outer variable is not modified — it resumes after the block ends.
    Block 4 — Under the Hood

    How Does JavaScript Actually Run?

    • Everything runs inside an Execution Context
    • JS is synchronous and single-threaded
    • One command at a time, in order

    Two phases

    1. Memory Creation Phase

    JS scans the code and allocates memory:
    • Variables (var) → set to undefined
    • Functions → stored entirely in memory

    2. Code Execution Phase

    JS runs code line by line. Variables get their actual values. Function calls create new execution contexts.

    Trace this example

    var x = 10;

function double(n) {
    var result = n * 2;
    return result;
}

var output = double(x);
    Memory Phase (Global EC):
    x → undefined
    double → ƒ(n) { ... }
    output → undefined

    Execution Phase:
    x → 10
    double(10) → creates new EC
    output → 20
    Block 4 — Under the Hood

    Call Stack

    Also called Runtime Stack / Machine Stack. It tracks which function is currently executing.

    function first() {
    console.log("first");
    second();
    console.log("first done");
}

function second() {
    console.log("second");
    third();
}

function third() {
    console.log("third");
}

first();
    Call Stack (LIFO):

    Step 1: [Global]
    Step 2: [Global] → [first]
    Step 3: [Global] → [first] → [second]
    Step 4: [Global] → [first] → [second] → [third]
    Step 5: third returns → popped
    Step 6: second returns → popped
    Step 7: first returns → popped
    Step 8: Global EC → done
    LIFO = Last In, First Out. Each function call pushes a new context onto the stack. When a function returns, its context is popped off.
    Block 4 — Under the Hood

    Hoisting

    During the memory creation phase, JS allocates space for variables and functions before any code runs. This means some things are available before their line in the code — this behavior is called hoisting.

    Functions are fully hoisted

    getName();
console.log(x);
console.log(getName);

function getName() {
    console.log("KeyValue Software Systems");
}

var x = 10;
    "KeyValue Software Systems"
    undefined
    f getName() { ... }

    getName works because function declarations are fully hoisted. x is undefined because var is hoisted but the value 10 isn't assigned yet.

    Arrow function = NOT hoisted like functions

    getName(); // What happens?

var getName = () => {
    console.log("KeyValue Software Systems");
};
    TypeError: getName is not a function

    Arrow functions (and function expressions) assigned to var are hoisted as undefined — calling undefined() throws a TypeError.

    Block 4 — Under the Hood

    let / const with Hoisting & TDZ

    Predict this

    console.log(a);
console.log(b);

let a = 10;
var b = 20;
    ReferenceError: Cannot access 'a' before initialization

    let and const are hoisted, but they sit in the Temporal Dead Zone (TDZ) until their declaration is reached. Accessing them before that throws a ReferenceError.

    TDZ = The zone between the start of the scope and the actual let/const declaration. Accessing the variable here is an error.

    undefined vs "not defined"

    var x;
console.log(x);  // undefined (declared, no value)
console.log(y);  // ReferenceError: y is not defined

    JS is loosely typed

    let a = 10;
a = "hello";  // no error — type changed!
a = true;     // still no error
    Variables in JS don't have fixed types. A variable can hold a number, then a string, then a boolean. This is dynamic/loose typing.
